# 成员 C 系统集成与评测指南

适用角色：系统集成与评测负责人  
目标周期：第 1 天到第 7 天，后 3-5 天继续协助报告和答辩  
核心目标：把 A/B 的数据和模型结果接成统一实验流程，保证结果可复现、指标可信、图表能直接进入报告。

## 0. 先看这一段：白话解释

成员 C 不一定要写最多模型代码，但要保证全组结果能被串起来。你可以把自己理解成“实验总管”：A 给你文本结果，B 给你图像和融合结果，你负责用同一套标准算分、画图、记录命令，并把最终材料交给 D 和 E。

所谓评估，就是拿模型预测结果和标准答案对比。所谓消融实验，就是故意去掉某些模块，看效果会不会下降。例如只用文本、只用图像、文本加图像、全部特征分别跑一遍。

选修课项目不要求你搭建复杂平台。保底目标是：所有模型结果放进统一表格，能算出指标，能画出至少一张混淆矩阵和一张对比表。

## 0.1 保底方案 / 标准方案 / 拔高方案

| 方案     | 你要做什么                                      | 做到什么程度算合格                   |
| -------- | ----------------------------------------------- | ------------------------------------ |
| 保底方案 | 让 A/B 都输出 CSV，你用 scikit-learn 计算指标。 | 有统一指标表、混淆矩阵和错误案例。   |
| 标准方案 | 统一命令、目录、实验编号、图表输出。            | 报告组能按你的复现说明讲清实验流程。 |
| 拔高方案 | 自动生成所有指标表和图表。                      | 一条命令能更新最终实验结果和图表。   |

## 0.2 手把手步骤

1. 第 1 天先定文件格式，不要等模型写完：每个结果都必须有 `sample_id`、真实标签、预测标签、风险概率。
2. 建立实验记录表，哪怕先用 Markdown 表格也可以。
3. 要求 A/B 每次给结果时写清楚使用了哪份数据。
4. 第 3 天先用 A 的文本基线结果练习算指标。
5. 第 4 天接 B 的图像结果，确认测试集和 A 一样。
6. 第 5 天评估融合结果，把它和文本、图像、行为结果放在一张表里。
7. 第 6 天画图：混淆矩阵、指标柱状图、消融实验表。
8. 第 7 天整理交接包，让 D/E 不看代码也能讲清楚实验。

## 0.3 现成工具怎么选

| 场景         | 推荐选择               | 原因                         |
| ------------ | ---------------------- | ---------------------------- |
| 计算指标     | scikit-learn           | 已安装，函数成熟。           |
| 画图         | matplotlib 或 seaborn  | 适合画混淆矩阵和柱状图。     |
| 记录实验     | CSV 或 Markdown 表格   | 简单，不依赖额外平台。       |
| 结果很乱     | 统一 `sample_id` 对齐  | 避免不同模型样本对不上。     |
| 没有完整代码 | 手动汇总预测结果也可以 | 课程项目优先保证结果能解释。 |

## 0.4 失败时怎么办

| 问题               | 保底处理                                                 |
| ------------------ | -------------------------------------------------------- |
| A/B 输出格式不一样 | 先统一成 CSV，至少保留 `sample_id`、真实标签、预测标签。 |
| 指标很差           | 不要美化数据，如实展示，用错误案例解释原因。             |
| 融合模型没提升     | 写成重要发现：多模态效果依赖数据质量。                   |
| 没时间画 ROC 曲线  | 先画混淆矩阵和指标柱状图。                               |
| 命令跑不通         | 写清楚已完成的手动流程和输入输出，保住可解释性。         |

## 0.5 每天做到什么程度算合格

| 时间      | 合格线                            |
| --------- | --------------------------------- |
| 第 1 天   | 结果文件格式和目录规范已定。      |
| 第 2 天   | 实验记录表已建好。                |
| 第 3 天   | 能评估文本基线。                  |
| 第 4-5 天 | 能汇总文本、图像、融合模型结果。  |
| 第 6 天   | 至少有指标表、混淆矩阵和消融表。  |
| 第 7 天   | 交接包能让 D/E 直接写报告和 PPT。 |

## 0.6 可以直接写进报告/PPT 的说法

本项目由系统集成模块统一管理实验结果和评价指标，保证不同模型在同一测试集上进行比较。评价指标包括准确率、精确率、召回率和 F1 值，并通过混淆矩阵、消融实验和错误案例分析模型表现。即使最终融合模型提升有限，也可以通过统一评估说明不同模态在当前数据集中的贡献和局限性。

## 1. 你最终要交付什么

| 交付物         | 最晚时间 | 验收标准                                         |
| -------------- | -------- | ------------------------------------------------ |

| 实验记录表     | 第 2 天  | 每次实验有编号、输入、参数、指标和备注。         |
| 评估函数       | 第 3 天  | 能计算准确率、精确率、召回率、F1 值和混淆矩阵。  |

| 融合模型结果   | 第 5 天  | 能评估融合模型并输出指标。                       |
| 最终图表       | 第 6 天  | 混淆矩阵、ROC 或 PR 曲线、消融实验表能用于报告。 |
| 代码冻结交接包 | 第 7 天  | 报告组能按你的说明复述实验流程和结论。           |

## 2. 第 1 天：统一目录和命令

建议目录：

```text
data/raw/          # 原始数据，本地保存，不提交
data/processed/    # 清洗后的训练数据，本地保存，不提交
outputs/metrics/   # 指标表
outputs/figures/   # 图表
outputs/predictions/ # 预测结果
models/            # 模型权重，本地保存，不提交
src/               # 后续代码
```

建议约定命令格式：

```powershell
uv sync
uv run python src/prepare_data.py
uv run python src/train_text.py
uv run python src/train_image.py
uv run python src/train_fusion.py
uv run python src/evaluate.py
```

如果代码还没写完，也要先把命令命名规则定下来，避免后期每个人脚本名不一致。

### 当天检查清单

- A 知道清洗数据应该输出到哪里。
- B 知道图像向量应该输出到哪里。
- D 知道图表最终从哪里拿。
- E 知道 PPT 用图从哪里拿。

## 3. 第 2 天：建立实验记录表

建议字段：

| 字段     | 说明                            |
| -------- | ------------------------------- |
| 实验编号 | 例如 `exp_001`。                |
| 日期     | 实验运行日期。                  |
| 输入数据 | 使用的数据版本。                |
| 模型     | 文本、图像、行为或融合。        |
| 参数     | 主要训练参数。                  |
| 指标     | 准确率、精确率、召回率、F1 值。 |
| 输出路径 | 指标表和图表位置。              |
| 备注     | 错误、异常、改动原因。          |

实验记录表可以先用 Markdown 或 CSV，不要等代码完善后再记录。

## 4. 第 3 天：评估函数和基线汇总

### 必做指标

| 指标     | 作用                                 |
| -------- | ------------------------------------ |
| 准确率   | 总体正确率，但不能单独作为结论。     |
| 精确率   | 判断为风险内容时有多少是真的。       |
| 召回率   | 实际风险内容中有多少被找出来。       |
| F1 值    | 平衡精确率和召回率，适合类别不均衡。 |
| 混淆矩阵 | 展示误判和漏判。                     |

### 推荐输出格式

```text
model,accuracy,precision,recall,f1,notes
text_baseline,0.80,0.78,0.70,0.74,TF-IDF
image_baseline,0.72,0.69,0.61,0.65,ResNet18
fusion,0.85,0.82,0.79,0.80,text+image+behavior
```

### 当天检查清单

- 指标计算输入是预测标签和真实标签。
- 每个模型使用同一份测试集。
- 类别顺序没有写反。
- 表格能直接给 D 使用。

## 5. 第 4-5 天：接入文本、图像和融合结果

### 接入原则

1. 所有结果必须带 `sample_id`。
2. 所有模型必须在同一测试集上比较。
3. 缺失样本要记录，不要静默丢掉。
4. 同一个指标函数评估所有模型。

### 融合结果检查

| 检查项   | 要求                                                 |
| -------- | ---------------------------------------------------- |
| 输入维度 | 文本、图像、行为特征拼接前维度清楚。                 |
| 数据划分 | 训练、验证、测试不能混用。                           |
| 预测输出 | 至少包含 `sample_id`、真实标签、预测标签、风险概率。 |
| 指标输出 | 和文本、图像基线放在同一张表中。                     |

## 6. 第 6 天：导出图表和错误案例

必须准备：

1. 总指标对比表。
2. 消融实验表。
3. 混淆矩阵图。
4. ROC 或 PR 曲线。
5. 训练损失曲线，如果有训练过程。
6. 3-5 个典型错误案例。

给 D 的图表要有标题、图注和一句话解释。给 E 的图表要尽量简洁，适合放 PPT。

## 7. 第 7 天：代码冻结交接包

### 交接包结构建议

```text
交接包/
├── 数据说明.md
├── 模型说明.md
├── 实验记录表.csv
├── 最终指标表.csv
├── 图表清单.md
├── 错误案例.md
└── 复现命令.md
```

### 交给 D/E 的说明必须包含

- 这次实验的最终结论是什么。
- 哪个模型效果最好。
- 为什么多模态有价值。
- 哪些结果不能过度解读。
- 数据和模型的局限性是什么。

## 8. 后 3-5 天协助内容

| 时间            | 你要做的事                             |
| --------------- | -------------------------------------- |
| 第 8 天         | 复核 D 写的实验流程是否准确。          |
| 第 9 天         | 复核 PPT 中的指标和图表是否对应正确。  |
| 第 10 天        | 准备老师可能追问的评估问题。           |
| 第 11-12 天可选 | 协助模拟答辩，解释实验可信度和局限性。 |

## 9. 常见问题和补救方案

| 问题                 | 补救方案                                     |
| -------------------- | -------------------------------------------- |
| A/B 输出格式不一致   | 要求都带 `sample_id`，先写转换脚本或转换表。 |
| 不同模型测试集不一样 | 重新对齐同一份测试集，否则不能直接比较。     |
| 指标看起来异常高     | 检查数据泄漏、重复样本、训练测试混用。       |
| 融合模型不如单模态   | 保留结果，写成消融分析和局限性。             |
| 图表太复杂           | 报告用详细图，PPT 用简化图。                 |

## 10. 每天结束前自查

- 今天的结果是否可复现。
- 是否记录了实验编号和参数。
- 是否能解释每个指标的含义。
- 是否把最终表格同步给 D。
- 是否把适合展示的图同步给 E。
- 是否避免把本地数据、模型权重和输出大文件提交到仓库。
