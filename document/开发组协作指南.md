# 开发组协作指南（保姆级）

适用对象：硬核开发组 成员 A / 成员 B / 成员 C  
用途：搞清楚**谁在什么时候等谁的交付、谁在什么时候把什么交给谁**  
预读条件：已读完自己的个人指南（A / B / C 各自的 `成员X...指南.md`）

> 本文不重复个人指南中的技术细节，只写"配合层面"。技术细节见各自的个人指南。

---

## 0. 速览：三个人到底怎么配合

```text
         ┌──────────────────────────────┐
         │  成员 A  数据+文本+行为      │
         │  产出：数据表、文本向量、    │
         │        行为特征、文本预测    │
         └──────┬──────────┬────────────┘
                │          │
        ┌───────┘          └───────────┐
        ▼                              ▼
┌───────────────┐              ┌───────────────┐
│  成员 B       │              │  成员 C       │
│  图像+融合    │──────────────▶│  集成+评测    │
│               │  融合预测    │               │
│  产出：图像   │              │  产出：指标表 │
│  向量、融合   │              │  图表、交接包 │
│  预测、结构图 │              │  复现命令     │
└───────────────┘              └───────────────┘
        ▲                              │
        │   文本向量、行为特征          │
        └──────────────────────────────┘
        (A 产出，B 消费)               (A/B 产出，C 消费，C 产出给 D/E)
```

**核心依赖链（按时间）：**

```text
Day1: 三人约定接口 ────────────────────── 基础前提
Day2: A产出数据(image_path) ──────────── B开始干活的前置条件
Day3: A产出TF-IDF基线预测 ────────────── C开始评估的前置条件
Day4: A产出文本向量 ──────────────────── B做融合的必要输入
      B产出图像向量 ──────────────────── C汇总单模态的必要输入
Day5: A产出行为特征 ──────────────────── B做融合的必要输入
      B产出融合预测 ──────────────────── C评估融合的必要输入
Day6: A+B产出消融材料 ────────────────── C画图出表的必要输入
Day7: 三人交付物汇总 ──────────────────── C打包交给D/E
```

## 2. 7 天并行作战地图

下面按天展开。格式：**等谁交付 → 今天做什么 → 今天交付给谁**。

### 第 1 天：统一约定（三人共同工作）

```
┌─────────────┬──────────────────────┬──────────────────────┬──────────────────────┐
│             │       成员 A         │       成员 B         │       成员 C         │

│ 今天做什么  │ ①定标签二分类        │ ①和A约定image_path   │ ①定目录结构           │
│             │   normal / risk      │   格式（相对路径）   │   data/raw/等         │
│             │ ②定字段表7列         │ ②定缺失图片处理：    │ ②定命令命名规则       │
│             │ ③和B确认image_path   │   零向量 or 跳过     │ ③定结果文件格式：     │
│             │   格式（相对路径）   │ ③和C确认图像向量     │   CSV必须含           │
│             │ ④和C确认数据文件格式 │   保存格式           │   sample_id、真标签   │
│             │   （CSV/JSONL）      │ ④确认图片尺寸224x224 │   预测标签、风险概率   │
│             │                      │                      │ ④建立实验记录表模板   │
├─────────────┼──────────────────────┼──────────────────────┼──────────────────────┤
│ 今天交付给  │ →B: image_path格式   │ →A: 缺失图片处理方案 │ →A: 目录结构、文件格式│
│             │ →C: 数据文件格式说明 │ →C: 图像向量保存格式 │ →B: 同上              │
│             │ →D: 数据来源说明头   │ →D: 图像分支架构位置 │ →D: 实验流程概述       │
├─────────────┼──────────────────────┼──────────────────────┼──────────────────────┤

└─────────────┴──────────────────────┴──────────────────────┴──────────────────────┘
```

> **第1天必须对齐的三件事（全组同步会，5分钟）：**
>
> 1. 标签方案：`normal` / `risk` 二分类
> 2. `image_path` 格式：统一用相对路径，如 `data/raw/images/001.jpg`
> 3. 结果文件格式：CSV，必须含 `sample_id`、`true_label`、`pred_label`、`risk_prob`

> **消融实验的 5 个组合，分工如下：**
>
> - A 负责提供：仅文本预测、仅行为预测、文本+行为预测
> - B 负责提供：仅图像预测、文本+图像预测、全融合（文本+图像+行为）预测
> - C 负责：让所有预测在**同一份测试集上**比较，算指标，画图

### 3.1 A → B 接口

| ---------------------- | -------------- | ----------- | ------------------------------------------------------- | ---------------------------------------- |
| 数据表（含image_path） | 第2天          | CSV         | `sample_id`, `image_path`, `label`, `split`             | `001, data/raw/img/001.jpg, risk, train` |
| 文本向量               | 第4天或第5天初 | CSV 或 .npy | `sample_id`, 向量（多列或一个数组）、维度说明           | `001, 0.12, -0.34, ...` (768列)          |
| 行为特征               | 第5天          | CSV         | `sample_id`, 各特征列（如 `post_freq`, `url_count` 等） | `001, 5.2, 12, 0.3, 1.5, ...`            |
| 预测结果               | 第3天起        | CSV         | `sample_id`, `true_label`, `pred_label`, `risk_prob`    | `001, risk, risk, 0.87`                  |

> **B 拿到 A 的数据后必须立刻验证：** `sample_id` 是否都能在自己的图片目录中找到对应图片？找不到的立刻告诉 A。

### 3.2 B → C 接口

| 接口           | 传递时间 | 格式                | 必须包含的列/信息                                    |
| -------------- | -------- | ------------------- | ---------------------------------------------------- |
| 图像单模态预测 | 第3-4天  | CSV                 | `sample_id`, `true_label`, `pred_label`, `risk_prob` |
| 融合预测       | 第5天    | CSV                 | 同上                                                 |
| 消融各组合预测 | 第6天    | CSV                 | 同上，外加一列标明是哪个消融组合                     |
| 图像向量维度   | 第3-4天  | 纯文本/口头         | 比如 `ResNet18 输出 512 维`                          |
| 模型结构图     | 第5-7天  | PNG 或 Mermaid 代码 | 标注输入/输出维度                                    |

### 3.3 A → C 接口

| 接口           | 传递时间 | 格式     | 必须包含的列/信息                                    |
| -------------- | -------- | -------- | ---------------------------------------------------- |
| 文本基线预测   | 第3天    | CSV      | `sample_id`, `true_label`, `pred_label`, `risk_prob` |
| BERT预测       | 第4天    | CSV      | 同上                                                 |
| 行为特征       | 第5天    | CSV      | 除 `sample_id` 外，至少3列行为特征                   |
| 消融各组合预测 | 第6天    | CSV      | 同上，外加组合标记                                   |
| 数据说明       | 第7天    | Markdown | 来源、样本量、字段说明、脱敏规则                     |

### 3.4 统一 CSV 格式强制约定

**所有预测结果 CSV 必须遵守：**

```csv
sample_id,true_label,pred_label,risk_prob,model_name
001,normal,normal,0.12,text_tfidf
002,risk,risk,0.89,text_tfidf
003,risk,normal,0.45,text_tfidf
```

| 字段         | 类型       | 说明                                                                 |
| ------------ | ---------- | -------------------------------------------------------------------- |
| `sample_id`  | string/int | 唯一标识，所有模型共用同一套                                         |
| `true_label` | string     | 只能是 `normal` 或 `risk`                                            |
| `pred_label` | string     | 同 `true_label` 取值                                                 |
| `risk_prob`  | float      | 0~1，预测为 `risk` 的概率                                            |
| `model_name` | string     | 标识是哪个模型的输出（如 `text_tfidf`, `image_resnet`, `fusion_v1`） |

> **如果这个格式不一致，C 无法统一评估。任何人产出预测 CSV 时请逐列对照此表。**

## 6. 常见阻塞与补救

### 场景一：A 找不到合适数据

```
症状：第2天结束时 A 没有可训练的 CSV
影响链：A阻塞 → B没有image_path → B无法读图片 → C无法评估

补救：
1. 退回到保底方案：用英文公开文本数据先跑 TF-IDF（HuggingFace有现成的spam数据集）
2. 图片用少量样例/占位图片先跑通流程
3. 行为特征用模拟数据（在报告中写"x特征用系统模拟数据"）
4. 重点保障 Day3 能出文本基线指标
```

### 场景二：A 的 BERT 跑不动

```
症状：第4天 BERT 太慢或显存不够
影响链：A没文本向量 → B无法做融合 → C无法评估融合

补救：
1. A 退回 TF-IDF 作为文本特征（已经有了，Day3已跑通）
2. 冻结 BERT 只提取向量，不做微调
3. B 先做 文本(TF-IDF) + 图像 融合，这是保底融合方案
4. 报告里写"尝试了 BERT 但因算力限制采用 TF-IDF 方案"
```

### 场景三：A 的文本向量和 B 的图像向量维度对不上，无法拼接

```
症状：B 拼接时提示 dimension mismatch
影响链：B 无法训练融合模型

补救：
1. A 先告诉 B 文本向量是多少维（比如 TF-IDF 可能是5000维，BERT 是768维）
2. B 在拼接前打印每个输入的 shape，确认维度
3. 如果维度差异太大（TF-IDF 5000维 vs 图像 512维），用 PCA 或简单降维
4. 或者退回"分数加权融合"，不需要维度对齐
```

### 场景四：B 发现大量 image_path 对应的文件不存在

```
症状：B 读图片时 FileNotFoundError 很多
影响链：B 图像分支数据不足

补救：
1. B 统计失败比例，告诉 A 和 C
2. 缺失图片用零向量填充（保留样本）
3. 如果失败超过 50%，图像分支降级为"辅助实验"，主报告靠文本和行为
4. 在交接包里记录图片缺失比例
```

### 场景五：融合后指标不如文本单模态

```
症状：文本模型 F1=0.80，融合模型 F1=0.78
影响链：感觉多模态"做了没用"

补救（**这是正常发现，不是失败！**）：
1. C 如实记录，这是消融实验的核心结论
2. B 分析原因：图片信号弱？图片和文本数据不匹配？
3. D 写入报告："在当前数据集上，图像分支未带来显著提升，
   可能原因是公开数据中图片信号有限，后续可在真实平台数据上验证"
4. 这不影响答辩，反而是"知道局限性"的加分项
```

### 场景六：C 发现不同模型的 sample_id 对不上

```
症状：A 的预测有1000条，B 的预测只有800条
影响链：无法在同一测试集上公平比较

补救：
1. 所有模型必须使用 C 指定的同一份测试集 CSV
2. C 从 A 的数据里抽出一份标准测试集，把 sample_id 列表发给 A 和 B
3. A 和 B 只对这份 sample_id 列表做预测
```
