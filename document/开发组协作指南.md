# 开发组协作指南（保姆级）

适用对象：硬核开发组 成员 A / 成员 B / 成员 C  
用途：搞清楚**谁在什么时候等谁的交付、谁在什么时候把什么交给谁**  
预读条件：已读完自己的个人指南（A / B / C 各自的 `成员X...指南.md`）

> 本文不重复个人指南中的技术细节，只写"配合层面"。技术细节见各自的个人指南。

---

## 0. 速览：三个人到底怎么配合

```text
         ┌──────────────────────────────┐
         │  成员 A  数据+文本+行为      │
         │  产出：数据表、文本向量、    │
         │        行为特征、文本预测    │
         └──────┬──────────┬────────────┘
                │          │
        ┌───────┘          └───────────┐
        ▼                              ▼
┌───────────────┐              ┌───────────────┐
│  成员 B       │              │  成员 C       │
│  图像+融合    │──────────────▶│  集成+评测    │
│               │  融合预测    │               │
│  产出：图像   │              │  产出：指标表 │
│  向量、融合   │              │  图表、交接包 │
│  预测、结构图 │              │  复现命令     │
└───────────────┘              └───────────────┘
        ▲                              │
        │   文本向量、行为特征          │
        └──────────────────────────────┘
        (A 产出，B 消费)               (A/B 产出，C 消费，C 产出给 D/E)
```

**核心依赖链（按时间）：**

```text
Day1: 三人约定接口 ────────────────────── 基础前提
Day2: A产出数据(image_path) ──────────── B开始干活的前置条件
Day3: A产出TF-IDF基线预测 ────────────── C开始评估的前置条件
Day4: A产出文本向量 ──────────────────── B做融合的必要输入
      B产出图像向量 ──────────────────── C汇总单模态的必要输入
Day5: A产出行为特征 ──────────────────── B做融合的必要输入
      B产出融合预测 ──────────────────── C评估融合的必要输入
Day6: A+B产出消融材料 ────────────────── C画图出表的必要输入
Day7: 三人交付物汇总 ──────────────────── C打包交给D/E
```

## 2. 7 天并行作战地图

下面按天展开。格式：**等谁交付 → 今天做什么 → 今天交付给谁**。

### 第 1 天：统一约定（三人共同工作）

|            | 成员 A                                                                                                                 | 成员 B                                                                                                                    | 成员 C                                                                                                                                |
| ---------- | ---------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------- |
| 等谁交付   | 无（第一天）                                                                                                           | 无（第一天）                                                                                                              | 无（第一天）                                                                                                                          |
| 今天做什么 | ①定标签二分类 normal / risk<br>②定字段表7列<br>③和B确认image_path格式（相对路径）<br>④和C确认数据文件格式（CSV/JSONL） | ①和A约定image_path格式（相对路径）<br>②定缺失图片处理：零向量 or 跳过<br>③和C确认图像向量保存格式<br>④确认图片尺寸224x224 | ①定目录结构 data/raw/等<br>②定命令命名规则<br>③定结果文件格式：CSV必须含 sample_id、真标签、预测标签、风险概率<br>④建立实验记录表模板 |
| 今天交付给 | →B: image_path格式<br>→C: 数据文件格式说明<br>→D: 数据来源说明头                                                       | →A: 缺失图片处理方案<br>→C: 图像向量保存格式<br>→D: 图像分支架构位置                                                      | →A: 目录结构、文件格式<br>→B: 同上<br>→D: 实验流程概述                                                                                |
| ⚠️阻塞风险 | 标签定义不清，后面全乱                                                                                                 | image_path格式不确定，B没法读图片                                                                                         | 目录结构不统一，后面文件到处散落                                                                                                      |

> **第1天必须对齐的三件事（全组同步会，5分钟）：**
>
> 1. 标签方案：`normal` / `risk` 二分类
> 2. `image_path` 格式：统一用相对路径，如 `data/raw/images/001.jpg`
> 3. 结果文件格式：CSV，必须含 `sample_id`、`true_label`、`pred_label`、`risk_prob`

**🌿 今日 Git 操作：初始化分支体系**（第1天不写业务代码，但要搭好 Git 骨架）

- [ ] 三人各自 `git branch` 显示 `* dev-x`
- [ ] 标签方案全员统一、image_path 格式全员统一、CSV 5列格式全员统一

**PR Review 流程（以 A 的 PR 为例）：**

```text
1. A 在 GitHub 点 "New pull request"，base=main ← compare=dev-a
2. A 填写 PR 标题和描述（粘贴指标数据）
3. A 在右侧 Reviewers 选 C
4. C 收到通知 → 检查代码和指标 → 点 "Approve" → 点 "Merge"
5. A 合并后切回 main 拉最新：git checkout main && git pull
6. A 切回 dev-a 继续工作：git checkout dev-a && git merge main
```

**✅ 第3天验收清单：**

- [ ] A 的 TF-IDF PR 已被 C review 通过并合并到 main

### 第 4-5 天：A 出 BERT+行为特征，B 出融合，C 汇总评估

**为什么今天 B 必须发起 PR？** B 的融合是**全组核心模块**，必须通过 PR 合并到 main 后，C 的 Day6 消融实验才能在统一的 main 代码上跑。否则各人代码分散在各自分支，C 无法汇总。

```powershell
# B 的 PR 合并后，所有人同步 main
git checkout main && git pull
git checkout dev-x && git merge main   # 把融合代码拉到自己的分支
```

## 6. 常见阻塞与补救

### 场景一：A 找不到合适数据

```
症状：第2天结束时 A 没有可训练的 CSV
影响链：A阻塞 → B没有image_path → B无法读图片 → C无法评估

补救：
1. 退回到保底方案：用英文公开文本数据先跑 TF-IDF（HuggingFace有现成的spam数据集）
2. 图片用少量样例/占位图片先跑通流程
3. 行为特征用模拟数据（在报告中写"x特征用系统模拟数据"）
4. 重点保障 Day3 能出文本基线指标
```

### 场景二：A 的 BERT 跑不动

```
症状：第4天 BERT 太慢或显存不够
影响链：A没文本向量 → B无法做融合 → C无法评估融合

补救：
1. A 退回 TF-IDF 作为文本特征（已经有了，Day3已跑通）
2. 冻结 BERT 只提取向量，不做微调
3. B 先做 文本(TF-IDF) + 图像 融合，这是保底融合方案
4. 报告里写"尝试了 BERT 但因算力限制采用 TF-IDF 方案"
```

### 场景三：A 的文本向量和 B 的图像向量维度对不上，无法拼接

```
症状：B 拼接时提示 dimension mismatch
影响链：B 无法训练融合模型

补救：
1. A 先告诉 B 文本向量是多少维（比如 TF-IDF 可能是5000维，BERT 是768维）
2. B 在拼接前打印每个输入的 shape，确认维度
3. 如果维度差异太大（TF-IDF 5000维 vs 图像 512维），用 PCA 或简单降维
4. 或者退回"分数加权融合"，不需要维度对齐
```

### 场景四：B 发现大量 image_path 对应的文件不存在

```
症状：B 读图片时 FileNotFoundError 很多
影响链：B 图像分支数据不足

补救：
1. B 统计失败比例，告诉 A 和 C
2. 缺失图片用零向量填充（保留样本）
3. 如果失败超过 50%，图像分支降级为"辅助实验"，主报告靠文本和行为
4. 在交接包里记录图片缺失比例
```

### 场景五：融合后指标不如文本单模态

```
症状：文本模型 F1=0.80，融合模型 F1=0.78
影响链：感觉多模态"做了没用"

补救（**这是正常发现，不是失败！**）：
1. C 如实记录，这是消融实验的核心结论
2. B 分析原因：图片信号弱？图片和文本数据不匹配？
3. D 写入报告："在当前数据集上，图像分支未带来显著提升，
   可能原因是公开数据中图片信号有限，后续可在真实平台数据上验证"
4. 这不影响答辩，反而是"知道局限性"的加分项
```

### 场景六：C 发现不同模型的 sample_id 对不上

```
症状：A 的预测有1000条，B 的预测只有800条
影响链：无法在同一测试集上公平比较

补救：
1. 所有模型必须使用 C 指定的同一份测试集 CSV
2. C 从 A 的数据里抽出一份标准测试集，把 sample_id 列表发给 A 和 B
3. A 和 B 只对这份 sample_id 列表做预测
```

| B   | ResNet18 冻结特征提取 + 分数加权融合 | 拼接MLP写不出来时退回      |

**原则：先确保流程能跑通，再逐步升级到标准方案。拔高方案（OCR、注意力融合）绝不能阻塞保底交付。**

## 8. Git 分支协作规则

配合 README 中已定好的分支命名规则：

| 分支                        | 负责人 | 用途                                 |
| --------------------------- | ------ | ------------------------------------ |
| `main`                      | 全组   | 始终可运行的稳定版本，只通过 PR 合并 |
| `dev-a`                     | 成员 A | A 的长期开发分支                     |
| `dev-b`                     | 成员 B | B 的长期开发分支                     |
| `dev-c`                     | 成员 C | C 的长期开发分支                     |
| `feature/data-cleaning`     | A      | 数据清洗专项                         |
| `feature/text-baseline`     | A      | 文本基线专项                         |
| `feature/image-model`       | B      | 图像模型专项                         |
| `feature/multimodal-fusion` | B      | 多模态融合专项                       |
| `feature/evaluation`        | C      | 评估流程专项                         |
| `docs/report-outline`       | D      | 报告大纲                             |
| `docs/defense-slides`       | E      | PPT素材                              |

**日常流程：**

```text
每天开始:  git checkout dev-x  (自己的分支)
每天结束:  git add/commit/push dev-x
模块完成:  发起 PR → 另一位开发组成员 review → 合并到 main
```

**分支对应关系：**

- A 的代码合并到 main 前，PR 由 B 或 C review
- B 的代码合并到 main 前，PR 由 A 或 C review
- C 的代码合并到 main 前，PR 由 A 或 B review
- D/E 的文档 PR 由 C review（确保技术描述准确）
