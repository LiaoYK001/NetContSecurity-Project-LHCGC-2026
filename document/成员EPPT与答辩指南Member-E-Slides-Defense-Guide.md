# 成员 E PPT 与答辩指南

适用角色：PPT 制作、讲稿组织和答辩主讲  
目标周期：第 1 天到第 10 天，必要时延长到第 12 天  
核心目标：把报告中的研究问题、方法、实验和结论压缩成一套能讲清楚、能答辩、能展示小组成果的 PPT。

## 0. 先看这一段：白话解释

成员 E 的任务不是把报告全文塞进 PPT，而是把项目讲成老师能快速听懂的故事。你要回答四个问题：我们做什么问题，为什么这个问题难，我们用了什么办法，结果说明了什么。

你不需要讲清每一行代码，也不需要证明模型世界第一。选修课探索性项目更看重流程是否完整、分工是否清楚、结果是否能解释、遇到问题是否知道怎么分析。

如果实验效果不好，也不是答辩失败。只要你能说清楚原因，例如数据少、图片信号弱、类别不均衡、训练时间有限，并提出下一步改进，反而显得项目更真实。

## 0.1 保底方案 / 标准方案 / 拔高方案

| 方案 | 你要做什么 | 做到什么程度算合格 |
| --- | --- | --- |
| 保底方案 | 做 8 页 PPT：背景、问题、数据、方法、结果、局限、总结、分工。 | 能在规定时间内讲完整，不依赖现场运行代码。 |
| 标准方案 | 做 10-12 页 PPT，加入系统图、模型结构图、指标图和问答页。 | 老师能看懂项目流程和每个人贡献。 |
| 拔高方案 | 加入演示页、备用解释页、模拟问答记录。 | 被追问时能切到备用页解释数据、模型或指标。 |

## 0.2 手把手步骤

1. 第 1 天先问老师或同学：答辩几分钟，是否必须现场演示。
2. 先建空 PPT，不等实验结果。每页先写标题和一句话结论。
3. 背景页只讲痛点，不要堆大段文字。
4. 方法页用 `Architecture.mmd` 或系统流程图讲清楚 A/B/C 怎么配合。
5. 第 7 天前先做占位页：结果表占位、消融实验占位、错误案例占位。
6. 第 8 天拿 C 的图表替换占位内容。
7. 第 9 天写讲稿，每页只记 3 句话。
8. 第 10 天模拟答辩，让 A/B/C/D 每人问你至少 1 个问题。

## 0.3 不会技术时怎么讲

| 技术点 | 白话说法 |
| --- | --- |
| TF-IDF | 看哪些词更像垃圾广告或诈骗内容。 |
| BERT | 用现成中文语言模型理解一句话的上下文含义。 |
| ResNet | 用现成图片模型把图片变成可计算的特征。 |
| OCR | 把图片里的文字识别出来，再交给文本模型分析。 |
| 多模态融合 | 把文本、图片和账号行为放在一起综合判断。 |
| F1 值 | 同时考虑误报和漏报的综合分数。 |
| 消融实验 | 分别去掉某些模块，看它们有没有贡献。 |

## 0.4 失败时怎么办：结果不好时怎么讲

| 问题 | 答辩说法 |
| --- | --- |
| 指标不高 | 本项目是探索性实践，重点是跑通完整流程并分析问题来源。 |
| 多模态没有明显提升 | 当前数据中某些模态信号弱，消融实验帮助我们发现这一点。 |
| 没有现场演示 | 用流程图、截图和复现命令说明系统可以复现。 |
| 被问到模型细节 | 先用白话解释，再请对应开发成员补充。 |
| 被问到数据合规 | 强调公开数据集优先、脱敏处理、public 仓库不提交原始数据。 |

## 0.5 每天做到什么程度算合格

| 时间 | 合格线 |
| --- | --- |
| 第 1 天 | PPT 目录和答辩主线确定。 |
| 第 2-4 天 | 背景、问题、方法页完成。 |
| 第 5-7 天 | 结果页、图表页、错误案例页有占位。 |
| 第 8 天 | PPT 完成 60% 以上，填入初步结果。 |
| 第 9 天 | PPT 完整版和讲稿完成。 |
| 第 10 天 | 至少模拟答辩 1 次，问答表完成。 |

## 0.6 可以直接在答辩中说的话

我们的项目定位是选修课探索性实践，重点是围绕社交平台内容安全问题，跑通从公开数据、文本和图像特征提取、多模态融合到实验评估的完整流程。即使最终指标不是特别高，我们也通过消融实验和错误案例分析了原因，并提出了增加数据规模、加入 OCR 和优化融合方式等改进方向。

## 1. 你最终要交付什么

| 交付物 | 最晚时间 | 验收标准 |
| --- | --- | --- |
| PPT 目录 | 第 1 天 | 能覆盖问题、方法、数据、实验、结论和分工。 |
| PPT 初版 | 第 8 天 | 至少完成 60%，等待最终实验图表替换。 |
| PPT 完整版 | 第 9 天 | 页面完整，图表和报告一致。 |
| 讲稿 | 第 9-10 天 | 能在规定时间内讲完。 |
| 问答表 | 第 10 天 | 至少准备 10 个老师可能追问的问题。 |
| 模拟答辩记录 | 第 10-12 天 | 记录超时点、卡顿点和被问住的问题。 |

## 2. PPT 推荐结构

```text
1 研究背景与问题
2 任务定义
3 数据来源与预处理
4 系统总体流程
5 文本、图像、行为特征方法
6 多模态融合方法
7 实验结果与消融分析
8 错误案例与局限性
9 总结与展望
10 成员分工
```

如果答辩时间很短，压缩成 8 页左右；如果时间较长，可以扩展到 12-15 页。

## 3. 第 1 天：建立答辩主线

### 你要讲清楚的一句话

本项目针对社交平台垃圾广告、诈骗和有害内容难以仅靠文本识别的问题，尝试结合文本、图像和用户行为特征，构建一个可复现的多模态风险识别流程。

### 当天操作

1. 确认答辩时长。
2. 建立 PPT 目录。
3. 和 D 对齐报告目录。
4. 先做空白页框架，不等实验结果。
5. 列出需要开发组提供的图表。

### 当天检查清单

- PPT 有清晰开头和结尾。
- 每一页知道要讲什么。
- 已列出要向 A/B/C 要的素材。
- 已确认老师是否要求现场演示。

## 4. 第 2-4 天：先做背景和方法页

### 背景页

只讲三个点：

1. 有害内容会影响平台安全和用户体验。
2. 单纯文本检测容易被谐音、截图、变体图片绕过。
3. 多模态方法能利用更多信号。

### 方法页

建议画一张总流程图：

```text
公开数据
-> 文本清洗和文本模型
-> 图片预处理和图像模型
-> 用户行为特征
-> 多模态融合
-> 风险识别结果
```

不要在 PPT 堆太多公式。公式和细节放报告，PPT 重点讲逻辑。

## 5. 第 5-7 天：等结果时先准备页面占位

你可以先做这些页面：

| 页面 | 先放什么 |
| --- | --- |
| 数据样例 | 放脱敏样例，不放真实账号。 |
| 模型结构 | 先放流程图，等 B 给最终结构图。 |
| 指标页 | 先放空表格，等 C 给最终指标。 |
| 消融实验页 | 先放表格模板，等 C 给结果。 |
| 错误案例页 | 先放版式，等 A/B/C 给样例。 |

第 7 天开发组交接后，你要快速替换占位内容。

## 6. 第 8-9 天：完成 PPT 和讲稿

### 页面设计原则

| 原则 | 说明 |
| --- | --- |
| 一页一个结论 | 每页只表达一个核心观点。 |
| 图表少而清楚 | PPT 不放过密表格，复杂表格放报告。 |
| 标题说结论 | 不写“实验结果”，改写成“多模态融合提升了 F1 值”。 |
| 颜色克制 | 不要用过多颜色，重点突出最终方案。 |
| 不泄露隐私 | 图片样例必须脱敏，不能出现真实账号和链接。 |

### 讲稿写法

每页讲稿建议 3 句话：

1. 这一页讲什么。
2. 我们怎么做。
3. 这说明什么。

不要逐字背诵大段文字，重点记住逻辑顺序和关键数字。

## 7. 第 10 天：准备问答

至少准备以下问题：

| 问题 | 回答方向 |
| --- | --- |
| 为什么不用单纯文本模型？ | 文本容易被谐音、图片和变体绕过，多模态能补充信号。 |
| 数据是否合规？ | 使用公开数据集优先，爬虫只作为补充，且不采集隐私和登录后内容。 |
| 为什么选 BERT 和 ResNet？ | 成熟、稳定、可复现，适合短周期课程项目。 |
| 融合方式为什么简单拼接？ | 项目周期短，拼接 + MLP 稳定可解释，便于复现。 |
| 指标为什么看 F1 值？ | 类别可能不均衡，F1 值能综合精确率和召回率。 |
| 模型误判怎么办？ | 用错误案例分析局限性，并提出 OCR、更多数据、注意力融合等改进。 |
| public 仓库如何保护隐私？ | 不提交原始数据、密钥、账号、模型权重，只提交说明和脱敏样例。 |
| 如果没有 GPU 怎么办？ | 使用 TF-IDF、冻结预训练模型和小批量实验，保证流程跑通。 |
| 图像分支效果差是否失败？ | 不失败，消融实验能说明图像信号在当前数据中的贡献和限制。 |
| 后续如何改进？ | 扩大数据、加入 OCR、优化融合方式、接入合规真实平台样本。 |

## 8. 模拟答辩流程

1. 第一次试讲只看是否能讲完。
2. 第二次试讲记录每页用时。
3. 第三次试讲让 A/B/C/D 各问 2 个问题。
4. 把答不上来的问题写进问答表。
5. 最后压缩超时页面，保留关键结论。

## 9. 常见问题和补救方案

| 问题 | 补救方案 |
| --- | --- |
| 实验结果来得太晚 | 先完成所有页面结构，最后只替换数字和图表。 |
| PPT 内容太多 | 每页保留一个结论，细节放报告。 |
| 讲稿超时 | 删除背景冗余内容，重点讲方法和结果。 |
| 被问到技术细节 | 先回答核心逻辑，再请对应开发成员补充。 |
| 老师质疑数据合规 | 强调公开数据集优先、脱敏、public 仓库不提交原始数据。 |

## 10. 每天结束前自查

- PPT 是否和报告结论一致。
- 每张图是否有来源。
- 是否有页面文字过多。
- 是否能在规定时间内讲完。
- 是否准备了老师可能追问的问题。
- 是否没有出现真实账号、链接、二维码、手机号等隐私信息。

## 11. 最终提交清单

- PPT 终稿。
- 讲稿。
- 问答表。
- 演示备用文件。
- 成员分工页。
- 模拟答辩记录。
