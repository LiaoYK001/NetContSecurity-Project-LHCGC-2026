# 成员 D 报告撰写指南

适用角色：报告总编与排版统筹  
目标周期：第 1 天到第 10 天，必要时延长到第 12 天  
核心目标：把开发组的实验过程和结果写成结构清晰、格式统一、逻辑完整、可提交的课程报告。

## 1. 你最终要交付什么

| 交付物 | 最晚时间 | 验收标准 |
| --- | --- | --- |
| 报告目录 | 第 1 天 | 章节完整，能覆盖背景、方法、数据、实验、总结和分工。 |
| 绪论和相关技术草稿 | 第 3-4 天 | 不依赖最终实验结果也能先写。 |
| 数据和系统设计章节 | 第 5-7 天 | 能根据 A/B/C 的交接材料写清楚流程。 |
| 实验结果与分析 | 第 8-9 天 | 指标、图表、解释和局限性对应准确。 |
| 报告终稿 | 第 10 天 | 排版统一、图表编号正确、无明显错别字和技术错误。 |

## 2. 推荐报告结构

```text
1 绪论
2 相关技术
3 数据来源与预处理
4 系统总体设计
5 模型方法
6 实验设计与结果分析
7 总结与展望
8 成员分工
参考文献
附录
```

如果老师给了模板，以老师模板为准。没有模板时，先用这个结构保证完整性。

## 3. 第 1 天：建立报告框架

### 操作步骤

1. 向老师或课程群确认评分标准、字数、格式、提交文件类型。
2. 建立报告目录。
3. 建立图表编号规则，例如“图 1”“表 1”。
4. 建立参考文献记录表。
5. 和 E 对齐 PPT 结构，保证报告和 PPT 讲同一套故事。

### 当天检查清单

- 报告目录已发给全组确认。
- 已知道老师是否要求 Word、PDF 或 Markdown。
- 已建立图表编号规则。
- 已记录需要向开发组索要的材料。

## 4. 第 2-4 天：先写不依赖实验结果的章节

### 绪论写法

绪论建议回答四个问题：

1. 为什么社交平台内容风控重要。
2. 为什么垃圾广告、诈骗、有害内容难识别。
3. 为什么单纯文本检测容易被绕过。
4. 为什么要引入文本、图像和用户行为的多模态方案。

### 相关技术写法

按模块写，不要堆概念：

| 技术 | 写作重点 |
| --- | --- |
| 文本表示 | 说明 TF-IDF、BERT 或中文 RoBERTa 的作用。 |
| 图像特征 | 说明 ResNet 如何提取图片特征。 |
| 用户行为特征 | 说明发帖频率、互动比、重复率为什么能反映异常。 |
| 多模态融合 | 说明为什么把多种特征拼接后分类。 |
| 评价指标 | 说明准确率、精确率、召回率、F1 值的含义。 |

### 当天检查清单

- 绪论不夸大项目能力。
- 相关技术和本项目实际方法一致。
- 没有把未实现的方法写成已实现。
- 需要引用的来源已记录。

## 5. 第 5-7 天：接收开发组材料

### 向 A 索要

- 数据来源说明。
- 字段说明。
- 脱敏规则。
- 文本模型说明。
- 文本错误案例。

### 向 B 索要

- 图像预处理流程。
- ResNet 选择原因。
- 融合模型说明。
- 模型结构图。
- 图像或融合错误案例。

### 向 C 索要

- 最终指标表。
- 消融实验表。
- 混淆矩阵。
- ROC 或 PR 曲线。
- 复现命令。
- 实验局限性说明。

### 当天检查清单

- 每张图都知道来源。
- 每个指标都知道对应模型。
- 每段方法描述都让对应开发成员复核过。
- 缺失材料已列清单催开发组补齐。

## 6. 第 8-9 天：写实验结果与分析

### 实验章节推荐顺序

1. 实验设置：数据划分、评价指标、运行环境。
2. 单模态结果：文本、图像、行为特征分别表现。
3. 多模态融合结果：最终方案指标。
4. 消融实验：说明每种特征贡献。
5. 错误案例：说明模型失败原因。
6. 局限性：说明数据量、分布差异、OCR、真实场景差距。

### 结果分析写法

不要只写“效果很好”，要写清楚：

- 哪个模型最好。
- 比基线提升了多少。
- 提升可能来自哪里。
- 哪些类别容易误判。
- 项目还有哪些限制。

## 7. 第 10 天：排版和终稿

### 排版检查

| 项目 | 要求 |
| --- | --- |
| 标题层级 | 一级、二级、三级标题格式统一。 |
| 图表编号 | 图和表按出现顺序编号。 |
| 图表引用 | 正文中必须提到每张图和每张表。 |
| 字体字号 | 按老师模板或统一规则。 |
| 参考文献 | 格式统一，不要只贴链接。 |
| 成员分工 | 明确写出 A-E 的贡献。 |

### 技术复核

终稿前让 A/B/C 分别看自己负责的部分。你的任务是统一表达，不要擅自改技术结论。

## 8. 常见问题和补救方案

| 问题 | 补救方案 |
| --- | --- |
| 开发组没及时给结果 | 先写实验设置和方法章节，结果表留占位，明确催交清单。 |
| 图表太多 | 报告保留关键图，其他放附录。 |
| 指标不够好看 | 如实分析原因，强调流程完整、消融清楚、局限性诚实。 |
| 技术细节看不懂 | 要求对应成员写 300-500 字大白话解释。 |
| PPT 和报告不一致 | 每次更新报告结论后同步通知 E。 |

## 9. 每天结束前自查

- 今天新增内容是否能被开发组确认。
- 图表是否有编号和解释。
- 是否存在没有来源的数据或结论。
- 是否有夸大模型能力的表述。
- 是否把隐私和数据合规写清楚。

## 10. 最终提交清单

- 报告终稿。
- 图表源文件或截图。
- 成员分工表。
- 参考文献列表。
- 实验结果表。
- 可复现命令附录。
- public 仓库隐私说明。
