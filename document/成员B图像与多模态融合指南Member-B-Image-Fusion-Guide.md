# 成员 B 图像与多模态融合指南

适用角色：图像模型与多模态融合负责人  
目标周期：第 1 天到第 7 天  
核心目标：跑通图像分支，设计可落地的多模态融合方案，并提供模型结构、融合结果和消融材料。

## 1. 你最终要交付什么

| 交付物 | 最晚时间 | 验收标准 |
| --- | --- | --- |
| 图片输入规范 | 第 1 天 | 明确图片路径、格式、尺寸、缺失图片处理方式。 |
| 图像基线结果 | 第 3-4 天 | ResNet18 或 ResNet50 能独立输出预测结果或图像向量。 |
| OCR 可选方案说明 | 第 4 天 | 说明是否启用 OCR，不启用也要写原因。 |
| 多模态融合第一版 | 第 5 天 | 能把文本向量、图像向量、行为特征拼接并输出分类结果。 |
| 消融实验材料 | 第 6 天 | 至少支持文本、图像、行为、融合模型的对比。 |
| 第 7 天交接材料 | 第 7 天 | 模型结构图、融合方式说明、指标表、300-500 字技术解释。 |

## 2. 第 1 天：确定图像输入规范

### 操作步骤

1. 和 A 确认 `image_path` 字段如何填写。
2. 约定没有图片的样本如何处理：
   - 方案一：丢弃没有图片的样本，只做纯多模态子集。
   - 方案二：保留样本，图像向量用全 0 填充。
   - 推荐先用方案二，样本利用率更高。
3. 确认图片格式：优先支持 `jpg`、`jpeg`、`png`。
4. 统一图片尺寸：ResNet 常用 `224 x 224`。
5. 和 C 确认图像向量保存格式，例如 `sample_id + image_embedding`。

### 当天检查清单

- 已明确缺失图片处理策略。
- 已和 A 对齐图片路径字段。
- 已和 C 对齐图像向量输出格式。
- 已告诉 D 图像分支在系统架构中的位置。

## 3. 第 2 天：准备图片预处理流程

### 推荐流程

```text
读取图片路径
-> 检查文件是否存在
-> 转成 RGB
-> resize 到 224 x 224
-> normalize
-> 输入 ResNet
```

### 需要处理的异常

| 异常 | 处理方式 |
| --- | --- |
| 图片路径为空 | 使用零向量或标记缺失。 |
| 图片文件损坏 | 跳过并记录 `sample_id`。 |
| 图片尺寸过小 | resize 后保留，但记录风险。 |
| 图片中有二维码或隐私 | 不提交原图，只给脱敏说明。 |

## 4. 第 3-4 天：跑通图像基线模型

### 模型选择

| 情况 | 推荐模型 |
| --- | --- |
| 时间紧、电脑一般 | ResNet18 预训练特征提取。 |
| 有较多图片样本 | ResNet50 特征提取或微调分类头。 |
| 图片主要是截图文字 | 加 OCR，把识别出的文字交给 A 的文本分支。 |

### 推荐做法

1. 先冻结 ResNet 主体，只提取图像向量。
2. 在图像向量后接一个简单分类器。
3. 输出图像单模态指标。
4. 保存图像向量，供第 5 天融合使用。

### 当天检查清单

- 图像模型能处理至少一批样本。
- 图像向量维度已记录。
- 图像单模态指标已交给 C。
- 图片预处理流程已写成文字，交给 D。

## 5. OCR 可选路线

OCR 不是必做项，只有在图片中明显有广告字、截图字、二维码旁边文字时才做。

### 使用原则

| 情况 | 是否建议 OCR |
| --- | --- |
| 大量图片是纯照片 | 不建议，收益可能不高。 |
| 大量图片是广告海报 | 建议，文字信号很强。 |
| 时间不足 | 不建议，避免阻塞主线融合。 |
| OCR 环境安装失败 | 放弃 OCR，把它写成后续改进。 |

OCR 结果不要直接作为最终结论，应该作为额外文本字段交给 A，再进入文本模型。

## 6. 第 5 天：多模态融合第一版

### 输入

| 输入 | 来源 |
| --- | --- |
| 文本向量或文本预测概率 | 成员 A |
| 图像向量或图像预测概率 | 成员 B |
| 用户行为特征 | 成员 A |
| 标签和数据划分 | 成员 A/C |

### 推荐融合方式

优先做特征拼接：

```text
文本向量 + 图像向量 + 行为特征
-> 拼接成一个总向量
-> MLP 分类器
-> 输出风险概率
```

如果时间不够，使用分数加权：

```text
文本模型概率 * 0.5 + 图像模型概率 * 0.3 + 行为模型概率 * 0.2
```

### 当天验收标准

- 融合模型能跑完训练或推理。
- C 能拿到融合模型预测结果。
- D 能拿到一段清楚的融合方法说明。
- E 能拿到一张简洁的模型结构图。

## 7. 第 6 天：消融实验

需要配合 C 完成以下对比：

| 实验 | 目的 |
| --- | --- |
| 仅文本 | 验证文本内容是否有效。 |
| 仅图像 | 验证图片是否提供额外信息。 |
| 仅行为 | 验证账号异常特征是否有效。 |
| 文本 + 图像 | 验证双模态提升。 |
| 文本 + 图像 + 行为 | 最终方案。 |

你需要解释：融合模型提升来自哪里，失败样例可能是什么原因。

## 8. 第 7 天：交接包

### 交给 C

- 图像模型预测结果。
- 图像向量维度和保存格式。
- 融合模型预测结果。
- 消融实验所需的各组输入。

### 交给 D

- 图像预处理流程。
- ResNet 选择原因。
- 融合方式说明。
- 模型结构图说明。
- 300-500 字技术解释。

### 交给 E

- 一张模型结构图。
- 一张消融实验柱状图建议。
- 2-3 个图片误判案例解释。

## 9. 常见问题和补救方案

| 问题 | 补救方案 |
| --- | --- |
| 图片太少 | 保留图像分支作为辅助实验，最终强调文本和行为特征。 |
| 图片加载失败很多 | 记录失败比例，先用可读取样本跑通模型。 |
| 融合后效果没有提升 | 做消融分析，说明图像信号弱或样本分布不匹配。 |
| OCR 太慢或装不上 | 不阻塞主线，把 OCR 写入未来改进。 |
| 向量维度对不上 | 先打印每个输入维度，再统一拼接前的数组形状。 |

## 10. 每天结束前自查

- 图像输入是否能和 A 的数据表对齐。
- 图像向量是否能被 C 读取。
- 融合方法是否足够简单、可解释、可复现。
- 是否给 D/E 提供了能写报告和做 PPT 的图表或说明。
- 是否避免提交原始图片和可能含隐私的截图。
